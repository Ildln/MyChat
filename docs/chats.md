# Chats

## Текущий шаг chats flow

На этом шаге в MyChat уже работает базовый direct chat flow с перепиской внутри выбранного чата.

Сейчас пользователь может:

1. иметь подтвержденную дружбу с другим пользователем
2. создать `direct chat`
3. получить список своих чатов
4. отправить сообщение в чат через `chat_id`
5. получить историю сообщений выбранного чата

Групповые чаты и real-time через WebSocket будут добавлены на следующих шагах.

## POST /chats/direct

Создает личный `direct chat` между текущим пользователем и его другом.

Для запроса нужен Bearer token:

```http
Authorization: Bearer <jwt>
```

Тело запроса:

```json
{
  "user_id": 2
}
```

Пример успешного ответа:

```json
{
  "id": 1,
  "type": "direct",
  "created_at": "2026-03-09T12:00:00Z",
  "members": [
    {
      "id": 1,
      "username": "alice"
    },
    {
      "id": 2,
      "username": "bob"
    }
  ]
}
```

## GET /chats

Возвращает список чатов текущего пользователя.

Для запроса нужен Bearer token:

```http
Authorization: Bearer <jwt>
```

На этом шаге корректная работа поддерживается для `direct chat`.

Список строится по таблице `ChatMember`: пользователь видит только те чаты, где он является участником.

## GET /chats/{chat_id}/messages

Возвращает историю сообщений выбранного чата.

Для запроса нужен Bearer token.

Поведение:

- история доступна только участнику чата
- если чат не существует, возвращается `404`
- если пользователь не состоит в чате, возвращается ошибка доступа
- сообщения возвращаются в безопасном виде
- история сортируется от старых сообщений к новым

Пример успешного ответа:

```json
[
  {
    "id": 1,
    "chat_id": 1,
    "user_id": 1,
    "text": "Привет",
    "created_at": "2026-03-12T10:00:00Z"
  }
]
```

## POST /chats/{chat_id}/messages

Отправляет новое сообщение в выбранный чат.

Для запроса нужен Bearer token.

Тело запроса:

```json
{
  "text": "Привет"
}
```

Поведение:

- отправка доступна только участнику чата
- если чат не существует, возвращается `404`
- если пользователь не состоит в чате, возвращается ошибка доступа
- текст сообщения не должен быть пустым
- сообщение привязывается к `chat_id`
- для мягкого перехода старое поле `room` тоже заполняется техническим значением `chat:{chat_id}`

Пример успешного ответа:

```json
{
  "id": 1,
  "chat_id": 1,
  "user_id": 1,
  "text": "Привет",
  "created_at": "2026-03-12T10:00:00Z"
}
```

## Правила доступа

- все chat endpoint'ы требуют авторизацию
- пользователь может читать и писать только в те чаты, где он есть в `ChatMember`
- direct chat создается только между друзьями
- один и тот же direct chat между парой пользователей не дублируется

## Бизнес-правила текущего шага

- список чатов строится по `ChatMember`
- список сообщений строится по `chat_id`
- сообщения direct chat теперь работают через `chat_id`
- старый `room`-подход не удален резко и пока сохраняется для совместимости
- на этом этапе поддерживается только тип чата `direct`

Этот шаг завершает базовый backend flow личного чата: пользователь может создать чат, увидеть его в списке, отправить сообщение и прочитать историю.
