# Auth

## Текущий auth flow

Сейчас в MyChat действует минимальный, но уже цельный auth flow:

- `POST /auth/register` регистрирует нового пользователя
- `POST /auth/login` выполняет вход существующего пользователя
- `GET /auth/me` возвращает текущего авторизованного пользователя по JWT
- в обоих успешных ответах возвращаются:
  - `user_id`
  - `username`
  - `access_token`
  - `token_type`

Пароль хранится в базе не в открытом виде, а в поле `password_hash`.

Текущая цепочка использования выглядит так:

1. пользователь вызывает `POST /auth/register` и создает аккаунт
2. затем может входить через `POST /auth/login`
3. после этого может проверить текущего пользователя через `GET /auth/me` с Bearer token

## POST /auth/register

Создает нового пользователя и сразу возвращает JWT access token.

Тело запроса:

```json
{
  "username": "alice",
  "password": "secret123"
}
```

Поведение:

- делает `strip()` для `username`
- отклоняет пустой `username`
- отклоняет пустой `password`
- отклоняет повторяющийся `username`
- сохраняет хеш пароля в `password_hash`
- возвращает JWT для нового пользователя

Пример успешного ответа:

```json
{
  "user_id": 1,
  "username": "alice",
  "access_token": "<jwt>",
  "token_type": "bearer"
}
```

## POST /auth/login

Выполняет вход существующего пользователя по `username` и `password`.

Тело запроса:

```json
{
  "username": "alice",
  "password": "secret123"
}
```

Поведение:

- делает `strip()` для `username`
- отклоняет пустой `username`
- отклоняет пустой `password`
- больше не создает пользователя автоматически
- если пользователь не найден, возвращает `401`
- если пароль неверный, возвращает `401`
- при успешном входе возвращает JWT и базовую информацию о пользователе

Пример успешного ответа:

```json
{
  "user_id": 1,
  "username": "alice",
  "access_token": "<jwt>",
  "token_type": "bearer"
}
```

## GET /auth/me

Возвращает базовую информацию о текущем авторизованном пользователе.

Для запроса нужен заголовок:

```http
Authorization: Bearer <jwt>
```

Поведение:

- читает JWT из Bearer token
- если токен отсутствует, возвращает `401`
- если токен невалидный, возвращает `401`
- если пользователь из токена не найден в базе, возвращает `401`
- если все валидно, возвращает `id` и `username` текущего пользователя

Пример успешного ответа:

```json
{
  "id": 1,
  "username": "alice"
}
```

## Примечание по безопасности

Для случаев «пользователь не найден» и «неверный пароль» используется одинаковый ответ `401` с одной и той же ошибкой.

Это сделано специально, чтобы не раскрывать через API, существует ли пользователь с конкретным `username`.

Для `GET /auth/me` невалидный токен и токен на несуществующего пользователя тоже приводят к `401`, чтобы авторизация вела себя предсказуемо и безопасно.
