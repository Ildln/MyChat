# Friends

## Текущий шаг friends flow

На этом шаге в MyChat реализованы:

1. отправка заявки в друзья
2. просмотр входящих заявок
3. просмотр исходящих заявок
4. принятие заявки
5. отклонение заявки
6. просмотр списка друзей

После принятия заявки создается запись `Friendship`, которая фиксирует дружбу в нормализованном виде.

## POST /friends/requests

Создает новую заявку в друзья от текущего авторизованного пользователя.

Для запроса нужен Bearer token:

```http
Authorization: Bearer <jwt>
```

Тело запроса:

```json
{
  "to_user_id": 2
}
```

Пример успешного ответа:

```json
{
  "id": 1,
  "from_user_id": 1,
  "to_user_id": 2,
  "status": "pending",
  "created_at": "2026-03-09T12:00:00Z"
}
```

## Бизнес-правила текущего шага

- endpoint доступен только авторизованному пользователю
- нельзя отправить заявку самому себе
- нельзя отправить заявку несуществующему пользователю
- нельзя создать дубликат активной `pending`-заявки между теми же пользователями
- в ответе возвращается только безопасная информация о заявке

## GET /friends/requests/incoming

Возвращает только входящие заявки текущего пользователя.

Для запроса нужен Bearer token.

В ответе возвращается список заявок, где:

- `to_user_id` совпадает с текущим пользователем

## GET /friends/requests/outgoing

Возвращает только исходящие заявки текущего пользователя.

Для запроса нужен Bearer token.

В ответе возвращается список заявок, где:

- `from_user_id` совпадает с текущим пользователем

## POST /friends/requests/{request_id}/accept

Принимает входящую `pending`-заявку.

Поведение:

- обработать заявку может только тот, кому она адресована
- нельзя повторно принять уже обработанную заявку
- заявка переходит в статус `accepted`
- создается запись `Friendship`
- дружба хранится в нормализованном виде: меньший `user_id` идет в `user_a_id`, больший в `user_b_id`
- дубликат дружбы не создается

## POST /friends/requests/{request_id}/decline

Отклоняет входящую `pending`-заявку.

Поведение:

- обработать заявку может только тот, кому она адресована
- нельзя повторно отклонить уже обработанную заявку
- заявка переходит в статус `declined`

## GET /friends

Возвращает список друзей текущего пользователя.

Для запроса нужен Bearer token.

Список строится только из подтвержденных записей `Friendship`.

Поведение:

- endpoint доступен только авторизованному пользователю
- в ответе возвращаются только безопасные поля пользователя
- friendship корректно учитывается и для `user_a_id`, и для `user_b_id`
- если дружбы нет, возвращается пустой список

Пример успешного ответа:

```json
[
  {
    "id": 2,
    "username": "bob"
  }
]
```

## Бизнес-правила текущего шага

- `incoming` возвращает только входящие заявки текущего пользователя
- `outgoing` возвращает только исходящие заявки текущего пользователя
- принять или отклонить заявку может только адресат
- повторно обработать уже не-`pending` заявку нельзя
- после `accept` создается `Friendship`
- пара пользователей в `Friendship` хранится в нормализованном виде без дублей
- список друзей строится из `Friendship`

Этот шаг завершает базовый friends flow: пользователь может отправить заявку, обработать ее и увидеть подтвержденного друга в списке.
